# OSPF NSSA Reference (RFC 3101 — Not-So-Stubby Areas)

## Purpose

NSSA allows an OSPF stub-like area to import external routes locally. Standard stub areas block all Type 5 (external) LSAs. NSSA relaxes this: an ASBR inside the area generates Type 7 LSAs instead of Type 5. The ABR at the area border translates Type 7 → Type 5 for flooding into the backbone.

This solves a real problem: you want stub-area benefits (no external LSA flooding from outside) but still need to redistribute routes from a local protocol (e.g., a branch router running BGP or static routes).

## Type 7 LSA

Type 7 LSAs are structurally identical to Type 5 (AS External) LSAs but with a different LSA type code and restricted flooding scope:

| Property | Type 5 | Type 7 |
|----------|--------|--------|
| Generated by | Any ASBR | ASBR inside NSSA only |
| Flooding scope | Entire AS (except stub areas) | NSSA area only |
| Carries | External prefix, metric, metric type (E1/E2), forwarding address | Same fields as Type 5 |
| Translated? | No | Yes — ABR converts to Type 5 at area border |

A Type 7 LSA can carry E1 or E2 metric types, just like Type 5. The metric type is preserved during translation.

## The P-Bit (Propagate Bit)

The P-bit in a Type 7 LSA's options field controls whether the ABR should translate it to Type 5:

- **P-bit = 1**: Translate this Type 7 to Type 5 (propagate to the backbone). This is the default when the NSSA ASBR generates the LSA.
- **P-bit = 0**: Do NOT translate. The route stays confined to the NSSA area. Used when the ASBR explicitly does not want the route to leave the area.

The ABR only translates Type 7 LSAs that have the P-bit set. Type 7 LSAs with P-bit = 0 are silently ignored during translation.

## Translator Election

When an NSSA has multiple ABRs, only one performs the Type 7 → Type 5 translation. The election rule:

1. The ABR with the **highest Router ID** becomes the elected translator.
2. All other ABRs suppress translation — they do NOT generate duplicate Type 5 LSAs.
3. If the elected translator fails (goes down), the ABR with the next-highest Router ID takes over.

To check which ABR is the translator: `get_ospf(device, "details")` shows the translator role and Router ID on each ABR.

**Important**: Translation is performed by the elected translator ABR, not by the ASBR that originated the Type 7. The ASBR only generates Type 7; the ABR converts it.

## Type 7 → Type 5 Translation Rules

When the elected ABR translates a Type 7 to Type 5:

1. **Forwarding address**: If the Type 7 has a non-zero forwarding address, it is preserved in the Type 5. If zero, the ABR uses its own interface IP as the forwarding address.
2. **Metric and metric type**: Preserved exactly (E1 stays E1, E2 stays E2, metric value unchanged).
3. **Advertising router**: The Type 5 shows the ABR as the advertising router, not the original ASBR. This means `show ip ospf database` on backbone routers shows the ABR's Router ID, not the NSSA ASBR's.
4. **LSA age**: Reset to 0 in the translated Type 5.

## Default Route in NSSA

Unlike stub areas, NSSA does **NOT** automatically inject a default route. This is a critical difference:

| Area Type | Default Route Behavior |
|-----------|----------------------|
| Stub | ABR automatically injects Type 3 default (0.0.0.0/0) |
| NSSA | No automatic default — must explicitly configure `area X nssa default-information-originate` on the ABR |
| NSSA Totally Stubby | ABR injects Type 3 default (same as totally stubby) |

If devices in the NSSA area have no default route and no full routing table, they cannot reach external destinations. This is a common misconfiguration.

### Default Route Configuration by Vendor

- **Cisco IOS**: `area <id> nssa default-information-originate` under `router ospf`
- **Arista EOS**: `area <id> nssa default-information-originate` under `router ospf`
- **Juniper JunOS**: `area <id> nssa default-lsa default-metric <metric>` under `protocols ospf`
- **Aruba AOS-CX**: `area <id> nssa default-information-originate` under `router ospf`
- **MikroTik RouterOS**: `default-originate=always` on the OSPF instance or area
- **VyOS**: `set protocols ospf area <id> area-type nssa default-information-originate`

## NSSA + Totally Stubby (NSSA No-Summary)

Combining NSSA with `no-summary` creates the most restrictive area type:

- No Type 3 inter-area summaries from outside (only a single Type 3 default from ABR)
- No Type 5 external LSAs from outside
- Type 7 LSAs generated locally by NSSA ASBRs (redistributed routes)
- ABR translates Type 7 → Type 5 for the backbone

This is configured with `area X nssa no-summary` on the ABR. Internal routers only see: Type 1 (Router), Type 2 (Network), Type 7 (NSSA External), and a single Type 3 default.

## NSSA vs Stub Comparison

| Feature | Stub | NSSA |
|---------|------|------|
| Blocks Type 5 from outside | Yes | Yes |
| Allows local redistribution | No | Yes (via Type 7) |
| Default route injection | Automatic | Manual (requires explicit config) |
| ABR translation needed | No | Yes (Type 7 → Type 5) |
| Can combine with no-summary | Yes (totally stubby) | Yes (NSSA totally stubby) |

## Troubleshooting NSSA Issues

### Type 7 Not Generated
- Check `redistribute` config on the NSSA ASBR: `get_ospf(device, "config")`
- Verify the area is configured as NSSA on the ASBR (not just on the ABR)

### Type 7 Present but Type 5 Missing in Backbone
- Check translator election: `get_ospf(abr_device, "details")` — is this ABR the elected translator?
- Verify ABR has `area X nssa` (not `area X nssa no-redistribution`)
- If multiple ABRs, only the one with highest RID translates

### No Default Route in NSSA
- Verify `area X nssa default-information-originate` is configured on the ABR
- Check LSDB for Type 3 default (0.0.0.0/0): `get_ospf(device, "database")`

### P-Bit Issues
- Type 7 LSAs with P-bit = 0 are not translated — this is by design
- Check if the ASBR is suppressing propagation intentionally
